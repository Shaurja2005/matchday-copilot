## PROJECT BRIEF

You are building **MatchDay Copilot**, a GenAI-enabled web application that enhances stadium operations and fan experience during the FIFA World Cup 2026. This is a hackathon-style submission that will be evaluated on code quality, security, efficiency, testing, and accessibility, so treat every requirement below as mandatory, not optional.

### Chosen vertical

**Fan Navigation, Crowd Management & Multilingual Real-Time Assistance** — a unified conversational + dashboard experience that helps three personas:

1. **Fans** — get wayfinding, gate/seat directions, queue wait times, accessibility routes, transport options, and answers in their own language.
2. **Volunteers / Venue Staff** — get an operational dashboard with live crowd-density signals, incident triage suggestions, and AI-drafted responses to common fan queries.
3. **Organizers** — get a control-room view with AI-generated situation summaries and recommended actions (e.g., "reroute Gate C queue," "open overflow shuttle").

Design the whole system around this persona logic — do not build a generic chatbot.

## 1. Core Features to Implement

### A. Fan-Facing Assistant (chat + map UI)
- Conversational assistant (GenAI-powered) that answers natural-language questions: "Where's the nearest accessible restroom to Gate D?", "How do I get to the stadium from downtown without a car?", "Is it crowded near the north entrance right now?"
- Multilingual support: detect input language, respond in the same language (support at minimum English, Spanish, French, Portuguese, Arabic — reflecting host/visiting nations for 2026).
- Context-aware logic: assistant should factor in stated user context (e.g., "I use a wheelchair," "I'm traveling with kids," "I have a 3pm kickoff ticket, Section 214") and adjust its answers — this contextual reasoning is a core evaluation point, not just Q&A.
- Interactive stadium map with zone highlighting, accessible-route overlay, and simulated live queue/crowd-density indicators.
- "Explain this in simple terms" / accessibility reading mode (larger text, screen-reader-friendly markup, high-contrast toggle).

### B. Staff/Volunteer Dashboard
- Live (simulated) crowd density heatmap per gate/zone.
- AI-generated shift briefings: summarize current hotspots and suggested actions in plain language.
- Quick-reply GenAI drafts for staff responding to common fan questions in any supported language (staff types in English, AI translates/localizes reply).
- Incident log with AI-suggested severity triage and recommended next step (not autonomous action — always a human-in-the-loop suggestion, log this design choice explicitly).

### C. Organizer / Control-Room View
- Aggregated summary generated by GenAI from simulated live signals (entry rates, queue lengths, transport delays, weather).
- Natural-language query bar: "Which gates need extra staff in the next 30 minutes?" → AI reasons over the mock dataset and returns a recommendation with justification.
- Sustainability panel: AI-estimated crowd transport carbon footprint with suggestions (e.g., promote shuttle over rideshare when demand spikes).

### D. Real-Time Decision Support Logic
- Implement this as an explicit rules + GenAI hybrid: deterministic thresholds (e.g., queue > X minutes) trigger a GenAI call that generates the human-readable recommendation and rationale. Document this hybrid logic clearly in code comments and README — evaluators want to see *logical decision-making*, not just an LLM wrapper.

## 2. Technical Requirements

### Architecture
- **Frontend:** React + TypeScript, Tailwind CSS.
- **Backend:** Node.js (Express) or Python (FastAPI) — pick one and be consistent.
- **GenAI integration:** Use the Anthropic Claude API (or whichever LLM platform you have set up) via a server-side service layer — never call the LLM API directly from the frontend, and never expose API keys client-side. All prompts/system instructions live in a dedicated `prompts/` module, not inline in route handlers.
- **Data layer:** Use a mock/simulated dataset (JSON or SQLite) representing gates, zones, queues, transport routes, and multilingual FAQ content — clearly labeled as simulated data for demo purposes (no real FIFA operational data exists to use).
- **RAG (optional but recommended):** Lightweight vector store (e.g., local embeddings + cosine similarity, or a library like `chromadb`) over a small knowledge base of stadium policies/FAQs so answers are grounded rather than hallucinated.

### Security (High Impact evaluation area)
- All API keys/secrets in environment variables, never committed; include a `.env.example`.
- Input validation and sanitization on every user-submitted field (chat messages, staff notes).
- Rate limiting on the GenAI endpoint to prevent abuse/cost overrun.
- No PII stored beyond what's needed for the session; document data-handling assumptions in the README.
- Add a basic content-safety check on LLM output before displaying to fans (e.g., reject/flag unexpected or unsafe completions).

### Code Quality
- Clear folder structure: `/frontend`, `/backend`, `/backend/services/genai`, `/backend/data`, `/docs`.
- Type safety throughout (TypeScript types / Python type hints + Pydantic models).
- Consistent linting/formatting (ESLint+Prettier or Black/Ruff) with config files committed.
- Meaningful commit history via Git (not one giant commit) — set up the repo with logical incremental commits.

### Testing (High Impact evaluation area)
- Unit tests for backend logic (decision-support thresholds, language detection, data access).
- At least a few integration tests mocking the GenAI API response so tests don't require live API calls.
- A simple test for accessibility-relevant UI behavior (e.g., alt text presence, ARIA roles) using something like `@testing-library/react` + `jest-axe`.
- Include an npm/CLI script (`npm test` / `pytest`) so evaluators can run tests in one command.

### Accessibility (High Impact evaluation area)
- Semantic HTML, ARIA labels, full keyboard navigation, visible focus states.
- Color contrast meeting WCAG AA minimum.
- Screen-reader-tested chat interface (announce new AI messages via `aria-live`).
- Text scaling / high-contrast mode toggle (mentioned above) actually wired to a real settings context, not decorative.

### Efficiency
- Debounce/throttle chat input and map re-renders.
- Cache repeated GenAI queries where sensible (e.g., static FAQ answers) to reduce token spend and latency.
- Lazy-load map assets and dashboard charts.

## 3. UI/UX Direction — Football/World Cup Themed

> **If using Google Stitch + Antigravity via MCP:** generate the screens below in Stitch first using natural-language design prompts, then have the Antigravity agent explicitly invoke the Stitch MCP server to fetch the HTML/CSS/design tokens and integrate them — Antigravity will not reach for Stitch on its own unless the prompt tells it to. See Step 5 of the Build Order below for the exact instruction to give the agent.

Build a polished, tournament-grade UI inspired by the **look and feel of the FIFA World Cup 2026** without using official FIFA trademarks, logos, or licensed team/venue branding (these are protected IP and shouldn't be reproduced or scraped). Instead:

- **Color palette:** pitch green, gold/trophy accent, deep navy or black, white — evokes World Cup broadcast graphics without copying any specific official mark.
- **Motifs:** soccer ball iconography (as generic shapes/icons, e.g., from Lucide/Heroicons or open icon sets), pitch-line patterns, stadium-silhouette illustrations, subtle confetti/scoreboard animation accents.
- **Typography:** bold condensed headline font (e.g., a sports-scoreboard-style Google Font like "Anton" or "Oswald") paired with a clean readable body font (Inter).
- **Imagery:** source generic stadium/crowd/football photography from royalty-free sources (Unsplash, Pexels, Pixabay) for hero/background imagery — do not pull images of official team crests, the FIFA World Cup trophy, or copyrighted match photography. If web-sourcing assets, prefer open-licensed icon libraries (Lucide, Heroicons, Font Awesome free tier) and stock-photo APIs (Unsplash API) called at build time, with attribution noted in the README.
- **Components:** scoreboard-style stat cards for the dashboard, a stadium-map component with a "you are here" pin and zone color-coding by crowd density, a chat widget styled like a matchday info kiosk.
- **Dark/light mode** toggle, mobile-first responsive layout (majority of fans will use this on phones inside the stadium).
